# 候选框深色模式设计

## 概述

为 PinyinTSF 的候选框增加深色模式支持。
当 Windows 应用主题为深色时，候选框自动切换到与 Figma 稿一致的深色视觉；当系统主题为浅色时，继续沿用当前浅色样式。
本次实现只做“自动跟随系统主题”，但代码结构需要预留后续“手动覆盖主题”的扩展点。

## 目标

1. 候选框在 Windows 深色主题下显示深色版视觉。
2. 候选框在 Windows 浅色主题下保持现有浅色视觉。
3. 不改变候选框现有交互、候选项布局、尺寸缩放和 TSF 行为。
4. 后续可以在不重写绘制逻辑的前提下，新增“浅色 / 深色 / 跟随系统”的手动主题覆盖开关。

## 非目标

1. 本次不新增独立设置窗口，主题覆盖选项直接加入现有托盘设置对话框。
2. 本次不重做候选框尺寸体系，只复用现有 `SettingsManager` 中的布局参数。
3. 本次不处理系统主题变化通知的全局实时广播，只保证候选框在重新显示或设置重载时会刷新主题。

## 现状

当前候选框是 `D:\PinyinTSF\src\CandidateWindow.cpp` 中的 Win32 自绘弹窗。
它已经具备以下能力：

1. 双缓冲绘制。
2. 圆角容器、分割线、选中项高亮和阴影绘制。
3. 基于 `SettingsManager` 的统一尺寸、圆角、字体和缩放参数。
4. `Show()`、`ReloadSettings()` 时会重新计算大小并触发重绘。

当前问题是颜色与阴影参数被硬编码为浅色常量，无法根据系统主题切换。

## 设计输入

本次深色视觉以用户提供的 Figma 文件为准：

- 文件：`候选框深色`
- 关键节点：同一页面中的“深色一声 / 深色二声 / 深色三声 / 深色四声 / 深色五声”

这些节点表达的是同一套候选框样式，只是选中项位置不同。
因此实现上应保留当前“按 `_selectedIndex` 绘制选中项”的机制，不为五个状态分别做分支。

## 主题检测策略

### 自动跟随系统

候选框主题通过读取 Windows 注册表判断：

- 路径：`HKCU\Software\Microsoft\Windows\CurrentVersion\Themes\Personalize`
- 键：`AppsUseLightTheme`

规则如下：

1. 值为 `0`：视为深色主题。
2. 值为 `1`：视为浅色主题。
3. 读取失败或键不存在：回退为浅色主题。

选择 `AppsUseLightTheme` 的原因是它对应 Windows 的应用主题，而候选框本质上属于应用层 UI，不应跟 `SystemUsesLightTheme` 绑定。

### 刷新时机

为了避免必须重启输入法 DLL 才生效，本次在以下时机刷新主题状态：

1. `CCandidateWindow::Show()`
2. `CCandidateWindow::ReloadSettings()`

这样用户切换 Windows 主题后，只要候选框再次出现，或者设置变更触发重载，就能看到正确主题。

## 未来手动覆盖开关的预留

本次需要提供 UI 和持久化配置，并且代码层要保留一个显式的主题来源抽象。

建议引入三态主题模式概念：

1. `FollowSystem`
2. `Light`
3. `Dark`

本次托盘设置页直接提供这三种模式。
后续如果继续扩展设置项，只需要：

1. 在 `SettingsManager` 中保存主题模式；
2. 在候选框解析当前有效主题时优先检查手动模式；
3. 复用同一套浅色 / 深色调色表与绘制代码。

本次会把该模式写入现有 `HKCU\Software\PinyinTSF` 配置下，并让候选框内部的主题解析逻辑按“主题模式 -> 有效主题 -> 调色表”这个方向组织，避免未来再次拆重构。

## 设置页集成

当前设置页来自托盘图标点击后打开的现有对话框，不需要新增入口。
本次直接在原设置页中加入主题覆盖选项。

### 交互形式

推荐使用单选按钮，三项分别为：

1. 跟随系统
2. 浅色
3. 深色

原因是这是一个互斥选择，单选按钮比复选框或下拉框更直接，也更容易和现有小型设置页兼容。

### 默认值

默认值为“跟随系统”。
首次安装或旧版本升级后，如果注册表中不存在该项，设置页和运行时都应落到“跟随系统”。

### 预览行为

设置页中的预览区域要跟随当前对话框里的主题选择即时刷新：

1. 选择“跟随系统”时，预览按当前系统主题渲染。
2. 选择“浅色”时，预览固定渲染浅色版。
3. 选择“深色”时，预览固定渲染深色版。

这样用户不必先保存再去输入框里试，能够在设置页内直接看到效果。

## 视觉实现策略

### 总体原则

1. 保持当前候选框布局、候选项宽高、字体大小和缩放体系不变。
2. 只切换与视觉相关的颜色、阴影和必要的混合方式。
3. 深色与浅色使用同一套绘制路径，避免出现两套分叉渲染逻辑。

### 需要主题化的元素

候选框中至少以下元素需要按主题切换：

1. 整体背景。
2. 容器填充色。
3. 容器描边色。
4. 分割线颜色。
5. 主文本颜色。
6. 次文本颜色（序号）。
7. 选中项背景色。
8. 选中项描边色。
9. 阴影基色与阴影混合方式。

### 绘制结构

现有绘制顺序可以保留：

1. 绘制窗口底色。
2. 绘制容器阴影。
3. 绘制圆角容器。
4. 绘制拼音标题。
5. 绘制分割线。
6. 绘制候选项高亮与文字。

但深色模式下要注意一个差异：

当前阴影辅助函数 `BlendWithWhite()` 是把黑色按透明度往白底混合，这只适合浅色画布。
如果继续在深色底上向白色混合，阴影会发灰发雾，不符合深色稿。
因此阴影混合需要改成“按目标背景色混合”，而不是写死对白色混合。

## 代码结构设计

### 1. 候选框内部增加主题抽象

修改 `D:\PinyinTSF\src\CandidateWindow.h` 和 `D:\PinyinTSF\src\CandidateWindow.cpp`。

在 `CCandidateWindow` 内部增加轻量主题结构，例如：

1. 当前有效主题枚举（浅色 / 深色）。
2. 一组运行时颜色值。
3. 主题刷新方法。

职责边界如下：

1. `SettingsManager` 仍负责尺寸和缩放。
2. `CandidateWindow` 自己负责系统主题解析和绘制用调色表选择。
3. 不把主题颜色再塞进大量新的全局宏，避免继续扩大硬编码。

### 2. 主题刷新逻辑

新增内部方法，职责拆分为：

1. 从 `SettingsManager` 读取当前主题模式。
2. 如果当前模式是跟随系统，则读取系统注册表判断系统主题。
3. 根据有效主题重建 GDI 资源。

因为画笔、画刷依赖颜色，所以每次主题变化后需要释放并重建：

1. 容器边框笔。
2. 分割线笔。
3. 选中项边框笔。
4. 背景刷。
5. 容器刷。
6. 选中项刷。

字体不依赖主题，不需要因主题变化而重建，除非本次已有设置重载同时要求更新字体。

### 3. GDI 资源管理

当前 `Create()` 中一次性创建所有颜色相关 GDI 对象，这对主题切换不够灵活。
需要将颜色相关对象的创建提炼为单独方法，例如：

1. 释放旧颜色资源。
2. 按当前主题重建颜色资源。

这样 `Create()`、`Show()`、`ReloadSettings()` 都能安全复用。

### 4. SettingsManager 的最小改动

本次需要把主题模式持久化到 `SettingsManager`。
同时为了避免让 `CandidateWindow` 和 `SettingsDialog` 各自定义一套枚举，主题模式应有统一声明。

推荐做法是：

1. 在 `SettingsManager` 中新增主题模式枚举、读写接口和默认值。
2. 在 `SettingsManager::Load()` / `Save()` 中读写一个新的 `ThemeMode` DWORD。
3. 在候选框和设置页里都复用这套枚举。

原因是这样既能满足本次设置页需求，又能保持主题来源只有一份真值。

## 验证标准

### 自动验证

由于现有测试项目不覆盖候选框 GDI 绘制和系统主题读取，本次自动测试只能覆盖纯逻辑与构建通过：

1. 主题模式解析与默认回退逻辑有纯测试覆盖。
2. 构建主 DLL 成功。
3. 逻辑测试项目保持可编译。

### 手动验证

必须进行以下手动验证：

1. Windows 为浅色主题时，候选框仍显示现有浅色样式。
2. Windows 为深色主题时，候选框切换为深色样式。
3. 设置页里可以切换“跟随系统 / 浅色 / 深色”三种模式。
4. 预览区会随主题选择即时更新。
5. 候选框的选中项移动、数字选词、回车提交、Esc 取消不受影响。
6. 修改系统主题后，在“跟随系统”模式下再次触发候选框显示，视觉能跟随变化。
7. 缩放和字体大小设置不受影响。

### Figma 对照重点

手动对照 Figma 时优先检查这些点：

1. 容器主底色是否为深色，而不是简单灰化的浅色版本。
2. 分割线和容器描边是否足够轻，但不会在深底上消失。
3. 选中项高亮块是否与底色区分明确。
4. 主文本与次文本层级是否清楚。
5. 阴影是否在深色背景下仍自然，不出现发白、发灰的脏边。

## 风险与取舍

### 风险 1：阴影在深色背景下失真

这是本次最容易出问题的点。
如果仍沿用对白色混合的阴影算法，深色模式的结果会明显偏离设计稿。
因此实现时应优先把阴影混合从“对白色混合”改成“对当前背景色混合”。

### 风险 2：主题切换时 GDI 资源泄漏

如果主题刷新时重复创建笔刷但没有释放旧对象，会导致资源泄漏。
因此颜色资源必须集中释放与重建，不能散落在多个路径里各自创建。

### 风险 3：未来手动覆盖开关反向侵入

如果本次仍然把颜色写回宏常量，后续做手动主题覆盖时还得再拆一次。
所以本次即使不做设置页，也要把“当前有效主题 -> 调色表”这层抽象建出来。

## 成功标准

满足以下条件即可视为本次设计完成：

1. 候选框在深色系统主题下与 Figma 深色稿整体观感一致。
2. 浅色模式不回归。
3. 托盘设置页已经提供手动主题覆盖开关。
4. 代码层仍保持“主题模式 -> 有效主题 -> 调色表”的清晰边界。
